The SQL Cheatsheet

Everything you've learned, in one scrollable page. Bookmark it. Print it. Reference it during interviews.

01 Basics

SELECT

SELECT col1, col2
FROM table_name;

-- All columns
SELECT * FROM customers;

-- Distinct values
SELECT DISTINCT city FROM customers;

Always end statements with ;. Prefer named columns over *.

WHERE

SELECT * FROM products
WHERE price > 50
  AND category = 'Electronics';

-- Operators
=  !=  <  >  <=  >=
AND  OR  NOT  IN  BETWEEN  LIKE  IS NULL

LIKE uses % (any chars) and _ (single char).

ORDER BY · LIMIT

SELECT * FROM customers
ORDER BY age DESC, name ASC
LIMIT 10 OFFSET 20;

Clause order is fixed: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT.

NULL handling

-- Right way
WHERE manager_id IS NULL
WHERE manager_id IS NOT NULL

-- Replace NULL
COALESCE(col, 'unknown')
IFNULL(col, 0)

= NULL never works. Anything touching NULL becomes NULL.

02 Aggregates & grouping

Aggregate functions

COUNT(*)         -- all rows
COUNT(col)       -- non-NULL only
COUNT(DISTINCT col)
SUM, AVG, MIN, MAX
ROUND(AVG(price), 2)

All aggregates ignore NULL except COUNT(*).

GROUP BY · HAVING

SELECT customer_id, SUM(total_amount) AS spent
FROM orders
WHERE status = 'delivered'
GROUP BY customer_id
HAVING SUM(total_amount) > 100
ORDER BY spent DESC;

WHERE = filter rows. HAVING = filter groups (uses aggregates).

03 Joins

INNER JOIN

SELECT c.name, o.order_id
FROM customers c
INNER JOIN orders o
  ON c.customer_id = o.customer_id;

Only rows that match in both tables.

LEFT JOIN

SELECT c.name, o.order_id
FROM customers c
LEFT JOIN orders o
  ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;  -- never ordered

All left rows + matches; NULL fills the right side.

Self-join

SELECT e.name AS emp, m.name AS mgr
FROM employees e
LEFT JOIN employees m
  ON e.manager_id = m.employee_id;

Aliases mandatory. Great for hierarchies.

CROSS JOIN (Cartesian)

SELECT c.name, p.name
FROM customers c
CROSS JOIN products p;
-- 8 × 8 = 64 rows

Every combination. Explodes fast — be deliberate.

04 Subqueries & CTEs

Subquery in WHERE

SELECT name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products);

EXISTS

SELECT name FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.customer_id
);

Faster than NOT IN on large data.

CTE (WITH)

WITH top_spenders AS (
  SELECT customer_id, SUM(total_amount) AS total
  FROM orders
  GROUP BY customer_id
  HAVING SUM(total_amount) > 100
)
SELECT c.name, t.total
FROM customers c
JOIN top_spenders t ON c.customer_id = t.customer_id;

Read top-down. Reusable, readable, debuggable.

Recursive CTE

WITH RECURSIVE nums AS (
  SELECT 1 AS n
  UNION ALL
  SELECT n + 1 FROM nums WHERE n < 10
)
SELECT * FROM nums;

For trees, graphs, sequences. Always include a termination condition.

05 Window functions

Ranking

ROW_NUMBER() OVER (
  PARTITION BY category
  ORDER BY price DESC
) AS rn

RANK()       -- 1, 1, 3, 4
DENSE_RANK() -- 1, 1, 2, 3
NTILE(4)    -- quartiles

LAG / LEAD

LAG(total_amount) OVER (
  ORDER BY order_date
) AS previous_amount

LEAD(total_amount, 2) OVER (...)
-- second row ahead

Boundary rows return NULL.

Running totals

SUM(amount) OVER (
  ORDER BY order_date
) AS running_total

AVG(amount) OVER (
  ORDER BY order_date
  ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
) AS moving_avg_3

Window vs GROUP BY

GROUP BY collapses rows. Window adds a calculated column without collapsing.

-- Each product + average for its category
SELECT name, price, category,
  AVG(price) OVER (PARTITION BY category)
FROM products;

06 Conditional & sets

CASE WHEN

SELECT name, price,
  CASE
    WHEN price < 25 THEN 'budget'
    WHEN price < 75 THEN 'mid'
    ELSE 'premium'
  END AS tier
FROM products;

Conditional counting: SUM(CASE WHEN cond THEN 1 ELSE 0 END)

Set operators

SELECT city FROM customers
UNION       -- dedupe
SELECT city FROM suppliers;

UNION ALL   -- keep duplicates (faster)
INTERSECT   -- in both
EXCEPT      -- in A not B (MINUS in Oracle)

07 String & date functions

Strings

UPPER(name)  LOWER(name)
LENGTH(email)
CONCAT(a, b)  -- or  a || b
SUBSTRING(name, 1, 3)
TRIM(col)  REPLACE(col, old, new)
col LIKE '%@gmail.com'

Dates

CURRENT_DATE  NOW()
EXTRACT(YEAR FROM signup_date)
DATE_PART('month', signup_date)
DATEDIFF(d1, d2)   -- MySQL
date1 - date2          -- Postgres
STRFTIME('%Y', date)  -- SQLite

Math

ROUND(price, 2)
FLOOR(x)  CEIL(x)  ABS(x)
MOD(a, b)   -- or  a % b
POWER(x, 2)  SQRT(x)

Casting

CAST(col AS INT)
CAST('2024-01-15' AS DATE)
col::INT            -- Postgres shorthand

08 Write & DDL

INSERT

INSERT INTO customers (name, city, age)
VALUES
  ('Maya Iqbal', 'Karachi', 30),
  ('Ravi Patel', 'Mumbai', 42);

-- Insert from query
INSERT INTO archive
SELECT * FROM orders WHERE status = 'cancelled';

UPDATE · DELETE

UPDATE products
SET price = price * 1.1
WHERE category = 'Electronics';

DELETE FROM orders WHERE status = 'cancelled';

⚠️ Always include WHERE — or risk wiping the whole table.

CREATE TABLE

CREATE TABLE orders (
  order_id INT PRIMARY KEY,
  customer_id INT NOT NULL,
  order_date DATE DEFAULT CURRENT_DATE,
  total DECIMAL(10, 2),
  FOREIGN KEY (customer_id)
    REFERENCES customers(customer_id)
);

Index · View

CREATE INDEX idx_orders_customer
  ON orders(customer_id);

CREATE VIEW top_customers AS
SELECT customer_id, SUM(total_amount) AS ltv
FROM orders GROUP BY customer_id;

09 Transactions

ACID transaction

BEGIN;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- On error:
ROLLBACK;

-- On success:
COMMIT;

Atomicity · Consistency · Isolation · Durability

Isolation levels

  • READ UNCOMMITTED — dirty reads possible
  • READ COMMITTED — default in PG, no dirty reads
  • REPEATABLE READ — default in MySQL, no phantom rows mostly
  • SERIALIZABLE — strongest, slowest, full safety

11 Performance

EXPLAIN

EXPLAIN SELECT * FROM orders WHERE customer_id = 12345;

-- With real timing:
EXPLAIN ANALYZE SELECT ...

Look for Seq Scan on big tables = missing index.

Top wins

  • Index columns used in WHERE, JOIN, ORDER BY
  • Avoid SELECT * — fetch only what you need
  • Filter early — push WHERE deep into subqueries
  • UNION ALL instead of UNION when dedupe not needed
  • Use EXISTS instead of IN on large subqueries
  • Composite index (a, b) serves filters on a or a AND b

Anti-patterns

  • ❌ WHERE UPPER(col) = 'X' — kills index
  • ❌ WHERE col LIKE '%abc%' — leading wildcard kills index
  • ❌ Functions on indexed column in WHERE
  • ❌ OR across different columns — often misses indexes
  • ❌ Implicit type casting in JOIN keys
  • ❌ N+1 queries — use a JOIN or batched IN
